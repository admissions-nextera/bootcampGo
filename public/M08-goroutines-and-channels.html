§1 The Problem Concurrency Solves

Imagine you need to fetch data from three different APIs. Done sequentially, you wait for API 1 to finish before starting API 2, and for API 2 before starting API 3. If each takes 300ms, you're waiting 900ms total.

But those three requests don't depend on each other. You could fire all three at the same time and be done in ~300ms. That's concurrency — doing multiple things at once instead of one after the other.

§2 Goroutines — Lightweight Concurrent Tasks

A goroutine is a function that runs concurrently with the rest of your program. You launch one with the go keyword in front of any function call. That's it. One word.

Goroutines are not OS threads — they're much cheaper. Go can run hundreds of thousands of goroutines in the same program, managing them with its own scheduler. A typical goroutine starts with just 2KB of stack space (vs ~1MB for a thread).

Goyour first goroutine
func sayHello(name string) {
    fmt.Println("Hello from", name)
}

func main() {
    // Normal call — runs synchronously
    sayHello("main")

    // Goroutine — runs concurrently, doesn't block
    go sayHello("goroutine 1")
    go sayHello("goroutine 2")
    go sayHello("goroutine 3")

    // ⚠️ Problem: main() might exit before goroutines finish!
    // We'll fix this in a moment.
}
⚠️
main() doesn't wait for goroutines

When main() returns, the entire program exits — even if goroutines are still running. If you run the code above, you might see zero, one, or all three goroutine messages, depending on timing. We need a way to wait for them. Enter: sync.WaitGroup.

WaitGroup — Waiting for Goroutines to Finish

A sync.WaitGroup is a counter. You add to it before launching a goroutine, and each goroutine calls Done() when it finishes. Wait() blocks until the counter hits zero:

Go sync.WaitGroup — annotated
var wg sync.WaitGroup
WaitGroup is a counter. Zero = all done. Must be zero-valued — don't initialize with make().
for i := 1; i <= 3; i++ {
Launch 3 goroutines.
wg.Add(1)
Increment counter BEFORE launching the goroutine. Never inside the goroutine — there's a race condition risk.
go func(id int) {
Anonymous goroutine. We pass i as a parameter (id) to avoid the classic loop closure bug.
defer wg.Done()
defer Done() — runs when this goroutine's function returns. Decrements the counter. The defer ensures it always runs even if the goroutine panics.
fmt.Printf("Worker %d done\n", id)
Do the actual work here.
}(i)
The (i) immediately invokes the function, passing i as the id parameter.
}
wg.Wait()
Block here until the WaitGroup counter reaches zero — meaning all goroutines called Done(). Only then does main() continue.
🐛
The loop closure bug — memorize this

If you write go func() { fmt.Println(i) }() inside a loop, every goroutine captures the same i variable. By the time they run, i is already at its final value. Always pass loop variables as parameters: go func(id int) { ... }(i). This is one of the most common Go gotchas.

§3 Channels — Communication Between Goroutines

"Don't communicate by sharing memory; share memory by communicating." — Go Proverb

Goroutines need to pass data to each other safely. The Go way is channels — typed conduits that let goroutines send and receive values. Think of a channel like a pipe: one goroutine puts values in one end, another reads from the other end.

Sender
ch ← value
→
chan int (buffered, cap=3)
42

—
→
Receiver

Gocreating and using channels
// Create a channel with make() — specify the type it carries
ch := make(chan int)       // unbuffered channel of ints
strCh := make(chan string) // unbuffered channel of strings

// Send a value into a channel — blocks until someone receives
ch <- 42

// Receive a value from a channel — blocks until someone sends
val := <-ch

// Close a channel — signals no more values will be sent
close(ch)

Unbuffered Channels — Synchronous Handoff

An unbuffered channel has no storage. A send blocks until a receiver is ready, and a receive blocks until a sender sends. It's a synchronous handoff — both sides must be present at the same time:

Gounbuffered channel — basic pattern
func main() {
    ch := make(chan string)

    // Launch a goroutine to send a value
    go func() {
        result := "Hello from the other goroutine!"
        ch <- result  // send — blocks until main() receives
    }()

    // Receive — blocks until the goroutine sends
    msg := <-ch
    fmt.Println(msg)
    // Hello from the other goroutine!
}

Buffered Channels — Async with a Buffer

A buffered channel has a capacity. Sends don't block until the buffer is full, and receives don't block as long as the buffer has values. Use them when the sender and receiver run at different speeds:

Gobuffered channels
// Buffered channel with capacity 3
ch := make(chan int, 3)

// These three sends don't block — buffer has room
ch <- 1
ch <- 2
ch <- 3

// ch <- 4  would block here — buffer is full

// Receive from the buffer
fmt.Println(<-ch)  // 1
fmt.Println(<-ch)  // 2
fmt.Println(<-ch)  // 3

// Range over a closed channel — receives all values then stops
results := make(chan int, 5)
for i := 1; i <= 5; i++ { results <- i * i }
close(results)

for v := range results {
    fmt.Println(v)  // 1, 4, 9, 16, 25
}

§4 Real Pattern — Fan-Out with WaitGroup + Channel

Here's the pattern you'll use constantly in real Go code: launch multiple goroutines to do work in parallel, collect all their results through a channel, and process them when done.

This is called fan-out — one goroutine fans out work to many, then collects results back in. Let's simulate fetching URLs concurrently:

Gofan-out pattern — parallel work + result collection
package main

import (
    "fmt"
    "sync"
    "time"
)

type Result struct {
    URL      string
    Duration string
    Status   string
}

// Simulates fetching a URL — takes variable time
func fetchURL(url string, delay time.Duration, results chan<- Result) {
    start := time.Now()
    time.Sleep(delay)  // simulate network latency
    results <- Result{
        URL:      url,
        Duration: time.Since(start).Round(time.Millisecond).String(),
        Status:   "200 OK",
    }
}

func main() {
    urls := []struct {
        url   string
        delay time.Duration
    }{
        {"https://api.forge.go/users",    120 * time.Millisecond},
        {"https://api.forge.go/products",  80  * time.Millisecond},
        {"https://api.forge.go/orders",    200 * time.Millisecond},
        {"https://api.forge.go/analytics", 60  * time.Millisecond},
    }

    // Buffered channel — one slot per URL so sends never block
    results := make(chan Result, len(urls))

    var wg sync.WaitGroup
    start := time.Now()

    // Fan-out: launch one goroutine per URL
    for _, u := range urls {
        wg.Add(1)
        go func(url string, delay time.Duration) {
            defer wg.Done()
            fetchURL(url, delay, results)
        }(u.url, u.delay)
    }

    // Close channel once all goroutines finish
    go func() {
        wg.Wait()
        close(results)
    }()

    // Collect results — range ends when channel is closed
    fmt.Println("Fetching all URLs concurrently...")
    for r := range results {
        fmt.Printf("  %-40s %s [%s]\n", r.URL, r.Status, r.Duration)
    }

    total := time.Since(start).Round(time.Millisecond)
    fmt.Printf("\nTotal time: %s (sequential would be ~460ms)\n", total)
}
terminalexpected output
Fetching all URLs concurrently...
  https://api.forge.go/analytics            200 OK [60ms]
  https://api.forge.go/products             200 OK [80ms]
  https://api.forge.go/users                200 OK [120ms]
  https://api.forge.go/orders               200 OK [200ms]

Total time: 202ms (sequential would be ~460ms)
💡
The results come back in completion order, not launch order

Analytics (60ms) finishes first even though it was launched fourth. This is concurrent execution — the Go scheduler runs all four goroutines simultaneously and your channel collects them as they arrive. This is why the total time is ~200ms instead of ~460ms.

§5 select — Handling Multiple Channels

select is like a switch statement for channels. It waits on multiple channel operations at once and runs the first one that's ready. If multiple are ready, it picks one at random:

Goselect — waiting on multiple channels
func main() {
    ch1 := make(chan string)
    ch2 := make(chan string)

    go func() {
        time.Sleep(100 * time.Millisecond)
        ch1 <- "result from service A"
    }()

    go func() {
        time.Sleep(200 * time.Millisecond)
        ch2 <- "result from service B"
    }()

    // Wait for whichever channel sends first
    for i := 0; i < 2; i++ {
        select {
        case msg := <-ch1:
            fmt.Println("ch1:", msg)
        case msg := <-ch2:
            fmt.Println("ch2:", msg)
        }
    }
    // ch1: result from service A   ← arrives at 100ms
    // ch2: result from service B   ← arrives at 200ms
}

Timeout Pattern with select

One of the most useful select patterns is implementing a timeout — give up waiting if a result doesn't arrive within a deadline:

Gotimeout with select + time.After
func fetchWithTimeout(ch <-chan string) {
    select {
    case result := <-ch:
        fmt.Println("Got result:", result)

    case <-time.After(500 * time.Millisecond):
        // time.After returns a channel that sends after the duration
        fmt.Println("⏱ Timeout! Service took too long.")
    }
}
👀
Directional channel types

Notice chan<- string (send-only) and <-chan string (receive-only) in the signatures above. These restrict what you can do with a channel parameter, making your intent explicit and catching mistakes at compile time. chan string is bidirectional and is what you create with make().

§6 Mutex — When You Must Share State

Channels are great for passing data between goroutines. But sometimes multiple goroutines need to access the same shared variable. Without coordination, this causes a data race — unpredictable, hard-to-debug bugs.

A sync.Mutex is a mutual exclusion lock. Only one goroutine can hold it at a time — others wait until it's released:

❌ Data race — broken
var counter int

for i := 0; i < 1000; i++ {
    go func() {
        counter++ // race! multiple
                  // goroutines read+write
                  // simultaneously
    }()
}
// Final value: unpredictable
✅ Mutex — safe
var (
    counter int
    mu      sync.Mutex
)
for i := 0; i < 1000; i++ {
    go func() {
        mu.Lock()
        counter++
        mu.Unlock()
    }()
}
// Final value: always 1000
💡
Always defer Unlock()

The idiomatic Go pattern is mu.Lock(); defer mu.Unlock() at the start of the protected section. defer guarantees the unlock happens even if the function panics or returns early — preventing a deadlock where every goroutine waits forever for a lock that's never released.

§7 Putting It All Together — Parallel Word Counter

Let's combine goroutines, WaitGroup, channels, and a Mutex in one real program — a parallel word counter that processes multiple documents simultaneously:

Goparallel_counter.go — type this and run it
package main

import (
    "fmt"
    "strings"
    "sync"
)

func countWords(doc string, totals map[string]int, mu *sync.Mutex, wg *sync.WaitGroup) {
    defer wg.Done()

    words := strings.Fields(doc)   // split on whitespace
    local := map[string]int{}       // local count — no lock needed

    for _, word := range words {
        word = strings.ToLower(strings.Trim(word, ".,!?"))
        local[word]++
    }

    // Merge local results into shared map — must lock
    mu.Lock()
    defer mu.Unlock()
    for word, count := range local {
        totals[word] += count
    }
}

func main() {
    documents := []string{
        "Go is fast and Go is simple",
        "Go has goroutines and Go is concurrent",
        "Simple is better than fast but Go is both",
    }

    totals := map[string]int{}
    var mu sync.Mutex
    var wg sync.WaitGroup

    for _, doc := range documents {
        wg.Add(1)
        go countWords(doc, totals, &mu, &wg)
    }

    wg.Wait()

    fmt.Println("Word frequencies:")
    for word, count := range totals {
        if count > 1 {
            fmt.Printf("  %-12s %d\n", word, count)
        }
    }
}
