Go doesn't have classes. It doesn't have inheritance. And it turns out — it doesn't need them. Structs hold your data. Methods define behaviour. Interfaces describe what something can do. Together, these three tools let you write clean, flexible, maintainable code that larger languages need a PhD to achieve.

§1 Structs — Custom Types That Hold Data

A struct is Go's way of grouping related data under a single custom type. You used one in Project #1 — the Task struct. Now let's understand exactly what's happening under the hood.

Godefining and using a struct
// Define a struct type — this is a blueprint, not an instance
type User struct {
    ID       int
    Name     string
    Email    string
    IsAdmin  bool
    Score    float64
}

func main() {
    // Create an instance — a struct literal
    u := User{
        ID:      1,
        Name:    "Anas",
        Email:   "[email protected]",
        IsAdmin: true,
        Score:   97.5,
    }

    // Access fields with dot notation
    fmt.Println(u.Name)     // Anas
    fmt.Println(u.Score)    // 97.5

    // Modify a field
    u.Score = 100
    fmt.Printf("%s is now at %.0f!\n", u.Name, u.Score)
    // Anas is now at 100!
}

Struct Zero Values & Partial Initialization

Any field you don't set gets its zero value automatically — just like regular variables. This means you can create partial structs safely:

Gopartial initialization
// Only set the fields you care about — rest get zero values
guest := User{Name: "Guest"}
fmt.Println(guest.ID)       // 0
fmt.Println(guest.Email)    // ""
fmt.Println(guest.IsAdmin)  // false

Pointers to Structs

By default, when you pass a struct to a function, Go makes a copy. If you want the function to modify the original, you pass a pointer to it using &:

Govalue vs pointer — the critical difference
// Value — function gets a COPY. Original unchanged.
func promoteByValue(u User) {
    u.IsAdmin = true   // only changes the copy
}

// Pointer — function gets the REAL thing. Original modified.
func promoteByPointer(u *User) {
    u.IsAdmin = true   // modifies the original
}

func main() {
    anas := User{Name: "Anas", IsAdmin: false}

    promoteByValue(anas)
    fmt.Println(anas.IsAdmin)   // false — copy was changed, not anas

    promoteByPointer(&anas)    // & takes the address of anas
    fmt.Println(anas.IsAdmin)   // true — anas was changed!
}
💡
Go auto-dereferences struct pointers

When you have a pointer to a struct (u *User), you'd normally need to write (*u).IsAdmin to access a field. But Go is smart — it lets you write u.IsAdmin directly and handles the dereferencing for you. This is why pointer methods feel identical to value methods at the call site.

§2 Methods — Behaviour Attached to Types

A method is just a function with a special receiver argument — it's associated with a specific type. This is how Go gives behaviour to structs without needing classes.

Go method anatomy — annotated
func (u User) Greet() string {
The (u User) part is the receiver. It says: "this method belongs to the User type, and inside this function, the instance is called u."
return fmt.Sprintf("Hi, I'm %s!", u.Name)
Access the struct's fields through the receiver variable u.
}
anas := User{Name: "Anas"}
Create an instance.
msg := anas.Greet()
Call the method using dot notation — just like accessing a field, but with parentheses. Reads naturally: "anas, greet yourself."

Value Receivers vs Pointer Receivers

Just like with functions, you choose between value and pointer receivers. The rule of thumb is simple: if the method needs to modify the struct, use a pointer receiver. If it only reads, a value receiver is fine.

Govalue vs pointer receivers
type Score struct {
    Value  int
    Player string
}

// Value receiver — reads only, no modification
func (s Score) Display() string {
    return fmt.Sprintf("%s: %d points", s.Player, s.Value)
}

// Pointer receiver — modifies the struct
func (s *Score) AddPoints(n int) {
    s.Value += n
}

func (s *Score) Reset() {
    s.Value = 0
}

func main() {
    s := Score{Player: "Anas", Value: 0}

    s.AddPoints(50)
    s.AddPoints(30)
    fmt.Println(s.Display())   // Anas: 80 points

    s.Reset()
    fmt.Println(s.Display())   // Anas: 0 points
}

The String() Method — Built-in Formatting Hook

If your type has a method called String() string, Go's fmt package will call it automatically whenever you print the value. This is Go's equivalent of toString() in other languages:

Gothe String() method
type User struct {
    Name  string
    Score float64
}

// fmt calls this automatically when printing a User
func (u User) String() string {
    return fmt.Sprintf("User(%s, score=%.1f)", u.Name, u.Score)
}

func main() {
    u := User{Name: "Anas", Score: 97.5}
    fmt.Println(u)    // User(Anas, score=97.5)  ← String() called automatically
    fmt.Printf("%v\n", u)  // same result
}

§3 Interfaces — Describing What Something Can Do

An interface is a contract. It says: "any type that has these methods satisfies this interface." You don't declare that a type implements an interface — if it has the right methods, it automatically qualifies. This is called implicit interface satisfaction, and it's one of Go's most elegant features.

"The bigger the interface, the weaker the abstraction." — Rob Pike

Go interfaces tend to be tiny — often just one or two methods. The most famous is the Stringer interface from the fmt package, which has exactly one method: String() string. That's it.

❌ Java / C# (explicit)
// Must explicitly declare
// "I implement this interface"
class Dog implements Animal {
    // Java forces you to say it
}
✅ Go (implicit)
// Just implement the methods.
// Go figures out the rest.
func (d Dog) Speak() string {
    return "Woof!"
    // If Animal needs Speak(),
    // Dog satisfies Animal. Done.
}

Defining and Using an Interface

Goa real interface example
// Define the interface — what behaviour do we need?
type Shape interface {
    Area()      float64
    Perimeter() float64
}

// Circle satisfies Shape — has both methods
type Circle struct { Radius float64 }

func (c Circle) Area()      float64 { return 3.14159 * c.Radius * c.Radius }
func (c Circle) Perimeter() float64 { return 2 * 3.14159 * c.Radius }

// Rectangle also satisfies Shape
type Rectangle struct { Width, Height float64 }

func (r Rectangle) Area()      float64 { return r.Width * r.Height }
func (r Rectangle) Perimeter() float64 { return 2 * (r.Width + r.Height) }

// This function accepts ANY type that satisfies Shape
func printShapeInfo(s Shape) {
    fmt.Printf("Area: %.2f | Perimeter: %.2f\n", s.Area(), s.Perimeter())
}

func main() {
    c := Circle{Radius: 5}
    r := Rectangle{Width: 4, Height: 6}

    printShapeInfo(c)  // Area: 78.54 | Perimeter: 31.42
    printShapeInfo(r)  // Area: 24.00 | Perimeter: 20.00
}

§4 The Empty Interface & Type Assertions

The empty interface interface{} (or its modern alias any) has zero methods — which means every type in Go satisfies it. It's the "accept anything" type, used when you genuinely don't know what type you'll receive:

Gothe empty interface
// any = interface{} — accepts literally any type
func describe(v any) {
    fmt.Printf("Value: %v | Type: %T\n", v, v)
}

func main() {
    describe(42)          // Value: 42      | Type: int
    describe("hello")     // Value: hello   | Type: string
    describe(true)        // Value: true    | Type: bool
    describe([]int{1,2}) // Value: [1 2]   | Type: []int
}

Type Assertions — Reclaiming the Concrete Type

When you have an any, you can assert the underlying type back using the .(Type) syntax. Use the comma-ok pattern to do it safely:

Gotype assertions — safe and unsafe
var v any = "Gopher's Forge"

// ❌ Unsafe — panics if wrong type
s := v.(string)
fmt.Println(s)   // works here, but dangerous

// ✅ Safe — comma-ok pattern (same as map lookups!)
s2, ok := v.(string)
if ok {
    fmt.Println("It's a string:", s2)
} else {
    fmt.Println("Not a string")
}

// Type switch — clean multi-type handling
func classify(v any) {
    switch t := v.(type) {
    case int:
        fmt.Printf("Integer: %d\n", t)
    case string:
        fmt.Printf("String: %q (len %d)\n", t, len(t))
    case bool:
        fmt.Printf("Boolean: %v\n", t)
    default:
        fmt.Printf("Unknown type: %T\n", t)
    }
}
⚠️
Use any sparingly

The empty interface turns off Go's type checker for that value. You lose all compile-time safety. Use it only when you genuinely need to handle multiple unknown types — like in JSON parsing or generic utility functions. In most cases, a concrete type or a specific interface is the right choice.

§5 Struct Embedding — Go's Answer to Inheritance

Go has no inheritance. But it has embedding — including one struct inside another. The outer struct automatically gets all the fields and methods of the embedded struct. It's composition instead of inheritance, and it's cleaner:

Gostruct embedding
type Base struct {
    ID        int
    CreatedAt string
}

func (b Base) Describe() string {
    return fmt.Sprintf("ID: %d, Created: %s", b.ID, b.CreatedAt)
}

type User struct {
    Base             // ← embedded — no field name, just the type
    Name  string
    Email string
}

type Product struct {
    Base             // ← also embedded in Product
    Title string
    Price float64
}

func main() {
    u := User{
        Base:  Base{ID: 1, CreatedAt: "2024-01-01"},
        Name:  "Anas",
        Email: "[email protected]",
    }

    // Access embedded fields directly — no Base. prefix needed
    fmt.Println(u.ID)           // 1 — promoted from Base
    fmt.Println(u.Name)         // Anas
    fmt.Println(u.Describe())  // ID: 1, Created: 2024-01-01 — promoted method!
}
🧠
Composition over inheritance — and it's better

In classical OOP, inheritance creates tight coupling — change the parent class and you risk breaking every child. Embedding is explicit. You can see exactly where each field and method comes from. It's like the Go compiler requires you to be honest about what you're reusing. Senior devs from Java and C++ often say Go's approach actually produces cleaner code.

§6 Putting It All Together

Let's build a small notification system using structs, methods, and interfaces — the kind of design pattern you'll see in real codebases:

Gonotifier.go — type this and run it
package main

import "fmt"

// Notifier interface — anything that can send a notification
type Notifier interface {
    Send(message string) error
    Name() string
}

// EmailNotifier satisfies Notifier
type EmailNotifier struct {
    ToAddress string
}

func (e EmailNotifier) Send(message string) error {
    fmt.Printf("📧 Email → %s: %s\n", e.ToAddress, message)
    return nil
}

func (e EmailNotifier) Name() string { return "Email" }

// SMSNotifier also satisfies Notifier
type SMSNotifier struct {
    PhoneNumber string
}

func (s SMSNotifier) Send(message string) error {
    fmt.Printf("📱 SMS → %s: %s\n", s.PhoneNumber, message)
    return nil
}

func (s SMSNotifier) Name() string { return "SMS" }

// sendAll takes any slice of Notifiers — works with Email, SMS, or anything else
func sendAll(notifiers []Notifier, message string) {
    fmt.Printf("Broadcasting: %q\n\n", message)
    for _, n := range notifiers {
        err := n.Send(message)
        if err != nil {
            fmt.Printf("  ❌ %s failed: %v\n", n.Name(), err)
        }
    }
}

func main() {
    notifiers := []Notifier{
        EmailNotifier{ToAddress: "[email protected]"},
        SMSNotifier{PhoneNumber: "+1-555-GOPHER"},
        EmailNotifier{ToAddress: "[email protected]"},
    }

    sendAll(notifiers, "Day 2 of Gopher's Forge has begun! 🔨")
}
terminalexpected output
Broadcasting: "Day 2 of Gopher's Forge has begun! 🔨"

📧 Email → [email protected]: Day 2 of Gopher's Forge has begun! 🔨
📱 SMS → +1-555-GOPHER: Day 2 of Gopher's Forge has begun! 🔨
📧 Email → [email protected]: Day 2 of Gopher's Forge has begun! 🔨
💡
This is the power of interfaces

sendAll doesn't know or care whether it's sending emails, SMS, Slack messages, or push notifications. As long as the type satisfies Notifier, it works. Adding a new notification channel later requires zero changes to sendAll — just define a new struct with Send() and Name() methods. This is extensible design the Go way.

🛠 Try It Yourself

Add a SlackNotifier struct to notifier.go with a Channel field (string). Implement the Notifier interface on it so it prints: 💬 Slack → #channel: message.

Add it to the notifiers slice in main() and run the program again. Notice that sendAll needed zero changes — it just worked. That's the interface payoff.
