§1 Anatomy of a Go Function

Functions in Go are defined with the func keyword. The syntax is clean and consistent — parameters and return types are clearly labeled, with no ceremony. Let's dissect one:

func keyword greet function name ( name string, age int ) parameters (name + type) string return type { ... }
Goa basic function
// func name(param type, param type) returnType { }
func greet(name string, age int) string {
    return fmt.Sprintf("Hey %s! You're %d and learning Go. Nice.", name, age)
}

func main() {
    message := greet("Anas", 25)
    fmt.Println(message)
}
// Output: Hey Anas! You're 25 and learning Go. Nice.
👀
fmt.Sprintf vs fmt.Println

fmt.Sprintf formats a string and returns it — it doesn't print anything. fmt.Println prints and returns nothing. Use Sprintf when you want to build a string to store or return. Use Println when you just want to see it.

Shorthand for Same-Type Parameters

When consecutive parameters share the same type, Go lets you write the type once at the end — a small but satisfying shorthand:

Goparameter shorthand
// Verbose: each param has its own type annotation
func add(a int, b int) int { return a + b }

// Shorthand: share the type across same-type params
func add(a, b int) int { return a + b }

// Works with multiple groups too
func describe(firstName, lastName string, age, score int) string {
    return fmt.Sprintf("%s %s — age: %d, score: %d", firstName, lastName, age, score)
}

§2 Multiple Return Values — Go's Superpower

Here's where Go diverges from almost every mainstream language: functions can return more than one value. This isn't a workaround or a hack — it's a first-class feature, and it fundamentally changes how Go handles errors.

Goreturning multiple values
// Return two values: result and a success flag
func divide(a, b float64) (float64, bool) {
    if b == 0 {
        return 0, false   // can't divide by zero
    }
    return a / b, true   // result + success signal
}

func main() {
    result, ok := divide(10, 3)
    if ok {
        fmt.Printf("Result: %.2f\n", result)  // 3.33
    }

    _, ok2 := divide(5, 0)
    if !ok2 {
        fmt.Println("Can't divide by zero!")
    }
}

Named Return Values

Go lets you name your return values in the function signature. This makes the function's intent crystal clear, and allows you to use a "naked return" — a return with no arguments that returns whatever the named values hold:

Gonamed return values
// Return values are named: 'sum' and 'product'
func calculate(a, b int) (sum int, product int) {
    sum = a + b         // assign to named return
    product = a * b     // assign to named return
    return              // naked return — returns sum and product
}

func main() {
    s, p := calculate(4, 5)
    fmt.Printf("Sum: %d, Product: %d\n", s, p)
    // Output: Sum: 9, Product: 20
}
⚠️
Use naked returns sparingly

Naked returns are fine in short functions, but in longer functions they hurt readability — a reader can't tell what's being returned without scrolling back to the signature. The Go community generally prefers explicit returns in anything longer than ~5 lines.

§3 Error Handling — The Go Way

This is the one that makes newcomers raise an eyebrow. In most languages, errors are handled with try/catch blocks — you write your happy path, then catch exceptions that might fly up from anywhere in the call stack.

Go doesn't have exceptions. In Go, errors are just values — returned from functions like any other return value, using the built-in error type. You check for them explicitly, right where they happen.

"Errors are values. Values can be programmed, and since you can program errors, there are many options for handling them." — Rob Pike, Go core team
🐍 Python / JavaScript
try:
    result = divide(10, 0)
    do_more_stuff(result)
    even_more(result)
except ZeroDivisionError as e:
    print(f"Error: {e}")

# Errors can come from ANYWHERE
# in the try block — hard to tell
# which line threw.
🐹 Go
result, err := divide(10, 0)
if err != nil {
    fmt.Println("Error:", err)
    return
}
// You KNOW result is safe here.
// No hidden control flow.
// Error source is 100% clear.

The error Type

error is a built-in interface in Go. It has one method: Error() string. You create errors with errors.New() or fmt.Errorf():

Gocreating and returning errors
import (
    "errors"
    "fmt"
)

// errors.New — simple static error message
var ErrDivisionByZero = errors.New("cannot divide by zero")

func divide(a, b float64) (float64, error) {
    if b == 0 {
        return 0, ErrDivisionByZero  // return zero value + the error
    }
    return a / b, nil              // nil means "no error"
}

// fmt.Errorf — dynamic error with context
func getUser(id int) (string, error) {
    if id <= 0 {
        return "", fmt.Errorf("invalid user id: %d", id)
    }
    return "Anas", nil
}

§4 Handling Errors Properly

Now let's look at the full picture of how to call a function that returns an error and handle it like a professional Gopher. This is the pattern you'll write hundreds of times — get it into your muscle memory.

Go the canonical error handling pattern — annotated
func processScore(score int) (string, error) {
Function signature: takes an int, returns a string AND an error. Convention: error is always the last return value.
if score < 0 || score > 100 {
Validate input. If something is wrong, we won't proceed.
return "", fmt.Errorf("score %d is out of range", score)
Return zero value for string ("") and a descriptive error. We never return a meaningful value alongside an error.
}
return fmt.Sprintf("Score %d accepted!", score), nil
Happy path: return the real value and nil for the error. nil means "everything is fine."
}
func main() {
msg, err := processScore(150)
Capture BOTH return values. Always. Even if you don't expect an error.
if err != nil {
The canonical error check. != nil means "an error occurred." Handle it here, immediately.
fmt.Println("Error:", err)
Log, return, or handle the error. Don't silently ignore it.
return
Return early. Don't let the code continue with a bad state — this is the "early return" pattern.
}
fmt.Println(msg) // safe to use here
By this line, you KNOW err is nil. msg is guaranteed to be valid. No defensive checks needed.
}
😅
Yes, you'll type "if err != nil" a lot

It's the most common complaint from Go newcomers. But here's the thing — every one of those checks is an explicit decision you made about what to do when something goes wrong. In try/catch languages, half of those decisions get silently skipped. Your 3am debugging sessions will be much shorter in Go.

§5 Variadic Functions — Accept Any Number of Arguments

A variadic function accepts a variable number of arguments of the same type. You've already been using one: fmt.Println accepts any number of arguments. Here's how to write your own:

Govariadic functions
// The ... before the type makes it variadic
func sum(numbers ...int) int {
    total := 0
    for _, n := range numbers {
        total += n
    }
    return total
}

func main() {
    fmt.Println(sum(1, 2, 3))           // 6
    fmt.Println(sum(10, 20, 30, 40))    // 100
    fmt.Println(sum())                  // 0 — zero args is valid

    // You can also spread a slice into a variadic function with ...
    scores := []int{88, 92, 76, 95}
    fmt.Println(sum(scores...))         // 351
}

§6 Functions Are First-Class Citizens

In Go, functions are values. You can assign them to variables, pass them as arguments, and return them from other functions. This unlocks powerful patterns like callbacks and function factories.

Functions as Variables

Gofunctions as values
func add(a, b int) int      { return a + b }
func multiply(a, b int) int { return a * b }

func main() {
    // Assign a function to a variable
    operation := add
    fmt.Println(operation(3, 4))  // 7

    operation = multiply
    fmt.Println(operation(3, 4))  // 12
}

Anonymous Functions & Closures

You can define a function inline without giving it a name. These are called anonymous functions. When an anonymous function references variables from its surrounding scope, it becomes a closure:

Goanonymous functions and closures
func main() {
    // Anonymous function — defined and called immediately
    result := func(x, y int) int {
        return x + y
    }(10, 20)
    fmt.Println(result)  // 30

    // Closure — captures 'base' from outer scope
    base := 100
    addToBase := func(n int) int {
        return base + n  // 'base' is captured from main()
    }
    fmt.Println(addToBase(5))   // 105
    fmt.Println(addToBase(20))  // 120
}
💡
You already used closures in M04

In the goroutine example from M01 (and our concurrent scraper in Day 2), we'll use anonymous functions with go func() { ... }(). That's exactly this pattern. Closures and goroutines are best friends in Go.

§7 Putting It All Together

Let's write a real utility package — a mini calculator with proper error handling. This uses everything from this module: functions, multiple returns, named returns, error handling, and variadic functions.

Gocalculator.go — type this out and run it
package main

import (
    "errors"
    "fmt"
)

// Sentinel error — a package-level named error
var ErrDivisionByZero = errors.New("cannot divide by zero")

// Basic arithmetic — multiple return values
func divide(a, b float64) (float64, error) {
    if b == 0 {
        return 0, ErrDivisionByZero
    }
    return a / b, nil
}

// Variadic sum — accepts any number of ints
func sumAll(nums ...int) int {
    total := 0
    for _, n := range nums {
        total += n
    }
    return total
}

// Named returns — documents what we're returning
func minMax(nums []int) (min int, max int, err error) {
    if len(nums) == 0 {
        err = errors.New("cannot find min/max of empty slice")
        return  // naked return: returns zero, zero, err
    }
    min, max = nums[0], nums[0]
    for _, n := range nums[1:] {
        if n < min { min = n }
        if n > max { max = n }
    }
    return  // naked return: returns min, max, nil
}

func main() {
    // Test divide
    result, err := divide(10, 3)
    if err != nil {
        fmt.Println("Divide error:", err)
    } else {
        fmt.Printf("10 / 3 = %.4f\n", result)
    }

    // Test divide by zero
    _, err = divide(5, 0)
    if err != nil {
        fmt.Println("Caught error:", err)
    }

    // Test variadic sum
    total := sumAll(10, 20, 30, 40, 50)
    fmt.Printf("Sum of all: %d\n", total)

    // Test minMax
    scores := []int{88, 42, 95, 67, 73}
    min, max, err := minMax(scores)
    if err != nil {
        fmt.Println("MinMax error:", err)
    } else {
        fmt.Printf("Min: %d, Max: %d\n", min, max)
    }
}
terminalexpected output
10 / 3 = 3.3333
Caught error: cannot divide by zero
Sum of all: 150
Min: 42, Max: 95
