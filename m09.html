§1 Packages — The Unit of Code Organization

A package is a directory of .go files that all share the same package name declaration at the top. Every Go file belongs to exactly one package. Packages serve two purposes: they organize your code into logical units, and they control what's visible to the outside world.

You've been using packages since day one. Every time you write fmt.Println, you're calling the Println function from the fmt package. fmt, os, strconv, sync — these are all packages from Go's standard library, and they work exactly the same way yours will.

📦
Package name vs import path

The package declaration and the import path are two different things. package mathutils is the name you use inside code. "github.com/anas/forge/mathutils" is the import path — the unique identifier used to find the package. The last segment of the import path is usually (but not always) the package name.

§2 Exported vs Unexported — Visibility by Convention

Go's visibility rule is beautifully simple: if a name starts with a capital letter, it's exported (public). If it starts with a lowercase letter, it's unexported (private). No public or private keywords. Just case.

NameExported?Accessible from other packages?
Add ✓ Yes Yes — capital A
add ✗ No No — lowercase a, package-private
User ✓ Yes Yes — capital U
userCache ✗ No No — lowercase u, internal implementation detail
ErrNotFound ✓ Yes Yes — exported sentinel error, consumers can check against it
errNotFound ✗ No No — internal error, nobody outside the package sees it

This applies to everything — functions, types, struct fields, variables, constants, and methods. It's enforced by the compiler. Try to access an unexported name from another package and you get a compile error. No runtime surprises.

Gomathutils/mathutils.go
package mathutils

import "errors"

// Exported error — consumers can compare against this
var ErrDivisionByZero = errors.New("division by zero")

// Add is exported — capital A. Usable from any package.
func Add(a, b int) int { return a + b }

// Divide is exported with proper error handling
func Divide(a, b float64) (float64, error) {
    if b == 0 {
        return 0, ErrDivisionByZero
    }
    return a / b, nil
}

// Average is exported
func Average(nums []float64) float64 {
    if len(nums) == 0 { return 0 }
    total := 0.0
    for _, n := range nums { total += n }
    return total / float64(len(nums))
}

// clamp is unexported — internal helper, not part of the public API
func clamp(val, min, max float64) float64 {
    if val < min { return min }
    if val > max { return max }
    return val
}
Gostringutils/stringutils.go
package stringutils

import "strings"

// Truncate shortens a string to maxLen, appending "..." if cut
func Truncate(s string, maxLen int) string {
    if len(s) <= maxLen { return s }
    return s[:maxLen] + "..."
}

// Slugify converts a string to a URL-safe slug
func Slugify(s string) string {
    s = strings.ToLower(s)
    s = strings.ReplaceAll(s, " ", "-")
    return s
}

// IsPalindrome checks if a string reads the same forwards and backwards
func IsPalindrome(s string) bool {
    s = strings.ToLower(s)
    runes := []rune(s)
    for i, j := 0, len(runes)-1; i < j; i, j = i+1, j-1 {
        if runes[i] != runes[j] { return false }
    }
    return true
}

§3 Importing Your Own Packages

To use a package you've written, import it using the full module path — the module name from go.mod + the subdirectory. Then call its exported names using the packagename.FunctionName() syntax:

Go main.go — importing and using local packages
package main
main.go must be package main — this is the entry point.
import (
"fmt"
Standard library package — no path prefix needed.
"github.com/anas/forge-app/mathutils"
Your local package. Path = module name + subdirectory name.
"github.com/anas/forge-app/stringutils"
Another local package. Go finds it in the stringutils/ folder.
)
func main() {
sum := mathutils.Add(10, 20)
Call exported function using packagename.FunctionName syntax.
fmt.Println(sum) // 30
result, err := mathutils.Divide(10, 0)
if err != nil {
fmt.Println("Error:", err)
}
_ = result
Use _ to discard unused variables — the compiler requires it.
slug := stringutils.Slugify("Hello World")
Using the stringutils package — fully qualified name.
fmt.Println(slug) // hello-world
}

§4 go.mod — Your Module's Identity Card

The go.mod file defines your module. It was created when you ran go mod init and it's the thing that makes your project a proper Go module — not just a folder with some files in it.

Go Module go.mod — annotated
module github.com/anas/forge-app
The module path — the unique name of this module. Used as the prefix for all import paths within this project.
go 1.22
The minimum Go version this module requires. Go is backward compatible, so this is rarely a problem.
require (
External dependencies appear here after go get.
github.com/some/library v1.2.3
Each dependency is pinned to an exact version. go.sum stores cryptographic hashes to verify integrity.

The Essential go Commands

CommandWhat it does
go mod init <path> Create a new module. Run once per project.
go get <package> Add an external dependency. Updates go.mod and go.sum automatically.
go mod tidy Remove unused deps, add missing ones. Run before committing.
go run . Run the main package in the current directory.
go build ./... Build all packages in the module. Finds compile errors everywhere.
go test ./... Run all tests in the module.
go vet ./... Run the static analyser. Catches bugs the compiler misses.
go doc <package> Read documentation for any package from the terminal.

Adding an External Dependency

Go's package ecosystem lives at pkg.go.dev. Adding a third-party package is one command:

terminaladding a real external package
# Add the popular chi router (we'll use this in M10)
$ go get github.com/go-chi/chi/v5

# go.mod is automatically updated:
require (
    github.com/go-chi/chi/v5 v5.1.0
)

# go.sum is also updated — cryptographic verification
# Never edit go.sum manually

# Remove deps you no longer use
$ go mod tidy
💡
go.sum is your security guarantee

The go.sum file contains cryptographic hashes of every dependency. When you build, Go verifies these hashes against a global checksum database. If anyone tampers with a package after you added it, the build fails. This is supply-chain security baked into the toolchain — and it's one of the reasons Go is trusted for critical infrastructure.

§5 The Standard Library — Your Best Friend

Go ships with one of the most complete standard libraries of any language. Before reaching for an external package, check if stdlib has what you need — it usually does. Here are the packages you'll use most in real Go development:

fmt

Formatted I/O — printing, scanning, string formatting.

Println, Printf, Sprintf, Errorf
net/http

HTTP client and server. Full-featured — no framework needed for most APIs.

ListenAndServe, HandleFunc, Get
encoding/json

JSON encoding and decoding. Works seamlessly with structs.

Marshal, Unmarshal, NewEncoder
os

Operating system interface — files, environment, args, exit.

Args, ReadFile, WriteFile, Getenv
strings

String manipulation — everything you need, no third-party lib required.

Contains, Split, TrimSpace, ToLower
strconv

Type conversions between strings and numeric types.

Atoi, Itoa, ParseFloat, FormatInt
time

Time, durations, timers, formatting, parsing.

Now, Sleep, Since, Format, Parse
sync

Concurrency primitives — WaitGroup, Mutex, Once, RWMutex.

WaitGroup, Mutex, Once
errors

Error creation, wrapping, and inspection.

New, Is, As, Unwrap
log

Simple logging with timestamps. Drop-in for fmt in production code.

Println, Printf, Fatal, Fatalf
math

Mathematical functions and constants.

Sqrt, Abs, Round, Pi, MaxInt
testing

Built-in testing framework. No external library needed.

T.Run, T.Error, T.Fatal, Benchmark
🧠
Go's stdlib philosophy — batteries included, but not bloated

Go's standard library covers exactly what most programs need: HTTP, JSON, crypto, compression, SQL drivers, templates, testing. What it deliberately doesn't include: an ORM, a web framework, a logging framework. These choices are intentional — the Go team curates stdlib carefully and doesn't add things that have active community debate. When you need an opinionated layer, reach for a third-party package. But try stdlib first.

§6 Five Rules for Good Package Design

Knowing how packages work is one thing. Knowing how to design them well is what separates readable Go from the other kind. Here are the five rules the Go community has converged on:

rulespackage design — the Go way
// 1. Name packages after what they provide, not what they do
//    Good: package user    Bad: package userhandler
//    Good: package auth    Bad: package doauth

// 2. Keep package names short, lowercase, one word
//    Good: package cache   Bad: package cacheManager
//    Good: package db      Bad: package database_utils

// 3. Don't repeat the package name in function names
//    Good: user.New()      Bad: user.NewUser()
//    Good: http.Get()      Bad: http.HTTPGet()

// 4. Small, focused packages beat large, kitchen-sink ones
//    A package that does one thing well is more reusable
//    than a package that does ten things adequately.

// 5. Package-level variables are fine for config and singletons,
//    but avoid mutable global state — it makes testing a nightmare.
👀
The internal/ directory — package-private packages

Go has a special convention: any package inside an internal/ directory can only be imported by code in the parent directory and its subdirectories. It's Go's way of marking packages as "not for external consumption." You'll see this in large codebases: myapp/internal/database can be used by myapp/ but not by other modules.
